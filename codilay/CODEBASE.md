## Overview
# Overview

CodiLay is a comprehensive tool designed for automated documentation of codebases using language models (LLM). It integrates with various components to manage file processing, configuration, prompt-based interactions, and document storage to streamline the generation of structured markdown documentation for developers. 

## Architecture

CodiLay operates primarily through a central processing loop managed by the `Processor` module. It interacts with other crucial components such as `LLMClient` for language model communications, `WireManager` for handling unresolved references, and `DocStore` for documentation storage and management.

### Key Components
- **Processor (`src/codilay/processor.py`):** Manages the main logic for processing files, generating documentation prompts, and resolving wires via LLM interactions.
- **LLMClient (`src/codilay/llm_client.py`):** Acts as an intermediary client with support for multiple LLM providers, handling rate limiting and token management.
- **WireManager (`src/codilay/wire_manager.py`):** Tracks references between files, resolving dependencies to integrate different parts of the documentation seamlessly.
- **DocStore (`src/codilay/docstore.py`):** Stores and manages the markdown documents created, ensuring coherent updates in line with source file changes.

## Tech Stack
CodiLay utilizes various technologies:
- **Python:** Core programming language
- **LLM Providers:** Integrates with multiple LLM providers for semantics and doc generation 
- **Git:** Supports version control and change tracking
- **Rich (UI Framework):** Enhances console outputs

## Integration and Entry Points
- **CLI (`codilay.py`):** Offers a command-line interface for users to invoke documentation processes, interacting with processors, and managing outputs.
- **Configuration (`codilay.config.json`):** Flexible settings management for project-specific needs, allowing easy integration of ignore patterns, chunking, and more.

## Planner Implementation
> File: `src/codilay/planner.py`

The `Planner` class in `planner.py` is responsible for determining the optimal processing order of files using an LLM (Language Model) interaction. It is capable of generating a processing plan that organizes files from a given file tree, previously generated markdown contents, list of files, and the current state of the agent.

### Key Components:
- **LLMClient**: Utilized for interacting with the language model to create an optimized ordering plan.
- **Prompts**: The `system_prompt` and `planning_prompt` functions are used to generate system and user prompts respectively, guiding the LLM in creating plans.

### Planning Process:
1. **plan Method**: Accepts a file tree, markdown contents, file list, and agent state. Utilizes prompts to interact with the LLM and decides the processing order.
2. **_validate_plan Method**: Ensures the generated plan is valid by checking the integrity of `order`, `parked`, `park_reasons`, and `skeleton` in the result.
    - Adjusts the order to include all files not explicitly ordered or parked.
    - Ensures any parked files have legitimate reasons and validates the structure of the suggested documentation skeleton.

### Configuration Parameters:
The class makes use of a configuration object that provides custom settings, such as the `entry_hint`, which are utilized in crafting the prompts.

### Role in CodiLay:
The Planner is pivotal in organizing the processing flow, particularly ensuring that markdown documentation is generated in an efficient manner, respecting both logical dependencies and user preferences as captured in `config` settings.

→ See [LLM Prompt Templates for CodiLay] for details on how prompts are structured to guide documentation processes.

## Interactive Menu System
> File: `src/codilay/menu.py`

### Function: `_is_back`

The `_is_back` function serves as a utility to determine if the user input corresponds to a command intended to navigate away or cancel the current operation within the interactive menu system. By examining the input string `value`, the function evaluates whether it matches one of the predefined exit commands including `"0"`, `"b"`, `"back"`, `"q"`, `"quit"`, or `"cancel"`. This functionality is crucial for ensuring that users can intuitively exit from menu options, facilitating smooth user experience in navigating the CLI environment.

The function returns a boolean: `True` if the input matches a cancellation or back command, and `False` otherwise. It is utilized by other functions such as `_int_prompt_with_back` to standardize the handling of such commands across the menu system.


_prompt_api_key function

The `_prompt_api_key` function is designed to interactively prompt the user for an API key associated with a specific provider. Upon invoking, it requests the user to input the API key for the given `provider` and handles edge cases such as user cancellation and empty input.

### Purpose and Functionality
- The function retrieves metadata for the provider using `PROVIDER_META[provider]` to fetch a user-readable label for the prompt.
- It uses the `Prompt.ask` method to securely acquire the API key from the user, displaying instructions and offering the cancellation option (inputting "0").
- The function checks whether the user input indicates a cancel operation via `_is_back` which returns "back" if true.
- Successful input leads to the API key being stored in the `settings` object using its `set_api_key` method, and a confirmation message is printed.
- If no key is provided, a warning message is displayed, and the function returns `None`.

### Integration and Usage
- This function plays a critical role in managing and securely storing user API keys, linking user input with configuration persistence.

See also: `Settings.set_api_key` for the storage mechanism implementation.

### Function: `_menu_help`

The `_menu_help` function is designed to provide users with comprehensive guidance on how to use the CodiLay tool. It displays a help panel with instructions and tips for getting started with CodiLay, usage examples of the CLI, and navigation tips within the interactive menu.

**Purpose and Functionality:**

- It utilizes `console.print` to display a formatted panel with rich text including commands, CLI usage examples, and additional resource links.

  - **Quick Start:** Steps to initialize and start using CodiLay.
  - **CLI Usage:** Examples of various commands users can execute.
  - **Navigation:** Instructions for navigation within the application.
  - **Settings:** Information on configuration files and API key persistence.

- The layout is enhanced for readability with bold and colored text styles, using the rich text formatting features of the `Panel` class.
- Once the help information is displayed, the function pauses for user acknowledgment using `_pause`.

**Integration and Usage:**

## CodiLay CLI Command Interface
> File: `src/codilay/cli.py`

The `CodiLayGroup` class is a custom Click command group used in the CodiLay CLI to manage potential ambiguities between subcommands and target paths. It enhances flexibility in command execution by auto-dispatching certain unrecognized commands to the `run` subcommand while correctly addressing subcommand names or direct paths.

Key functionalities:
- **Ambiguity Resolution**: The `CodiLayGroup` checks the first argument after its own options. If not a known subcommand and not an option, it implicitly assumes the command is a path, automatically prepending `run` to the arguments. This allows for commands like `codilay .` or `codilay /some/path` to function as intended without explicit `run` notation.
- **Interactive Mode**: When invoked without any subcommand or path, it opens an interactive menu for streamlined user interaction.

### _finalize_and_write Function
- **Purpose**: The `_finalize_and_write` function finalizes the documentation process by writing the aggregated outputs to disk, handling both the codebase documentation (`CODEBASE.md`) and metadata such as `links.json`.
- **Data Flow**: It operates in several stages:
  - Initializes the finalization phase by setting up a `Processor` instance.
  - Calls `processor.finalize()` to complete any remaining processing.
  - Collects open and closed wires from the `wire_mgr` and updates the `docstore` accordingly.
  - Generates and writes `CODEBASE.md` using `docstore.render_full_document()`.
  - Serializes documentation metadata to `links.json`, including timestamps and commit information.
  - Updates and saves the current application state.
  - Displays a UI summary of the finalization results.
- **Integration Points**: This function heavily interacts with other components like `Processor`, `wire_mgr`, `docstore`, `state`, and `ui` to ensure complete and correct documentation output.

### Data Flow
- **Command Resolution**: Utilizes Click's command resolution mechanism to determine whether to execute a subcommand directly or modify the command flow by prepending `run` when necessary. This ensures the seamless execution of commands in a user-friendly manner.

## LLM Prompt Templates for CodiLay
> File: `src/codilay/prompts.py`

The `src/codilay/prompts.py` file contains key large language model (LLM) prompt templates used within CodiLay for generating documentation. These prompts guide the LLM interactions based on different stages of processing files in a project.

### Functions:

- **system_prompt(config):** Constructs the system prompt with contextual notes and instructions provided by the user. This template is crucial for ensuring that the AI generates consistent and accurate documentation in the defined JSON format.
  
- **triage_prompt(file_tree, all_files, md_contents=None, notes='', instructions=''):** Produces a prompt used for the triage phase, helping to classify files into categories such as CORE, SKIM, or SKIP based on their importance and relevance for detailed documentation.

- **planning_prompt(file_tree, md_contents, files, entry_hint=None):** Used during the planning phase to order files for processing, creating a strategic documentation order aligning with project dependencies.

- **processing_prompt(file_path, file_content, relevant_sections, open_wires, section_index):** Crafts a prompt for the processing stage, integrating current documentation state and open wires to facilitate focused documentation generation.

- **finalize_prompt(section_index, open_wires, parked_summaries):** Finalizes the documentation by summarizing and cross-referencing previously generated content, ensuring all wires are resolved.

- **skeleton_prompt(file_path, skeleton_content, section_index, open_wires):** Assists in the initial documentation of large files by providing a skeleton structure for further detail passes.

- **detail_prompt(file_path, chunk_content, chunk_label, chunk_index, total_chunks, skeleton_section_id, skeleton_content, open_wires):** Used during detailed sequential processing of file chunks to enrich initially generated skeleton documentation with comprehensive details.

These prompts play an integral role in guiding the LLM through the documentation setup, ensuring the generated content is structured and comprehensive.

The `triage_prompt` is employed in the `triage.py` module for the smart triage phase to provide the LLM with context on file structures and intent for classification without exposing the file content. This makes it a cost-effective solution for project scanning.

## File Chunking and Boundary Detection
> File: `src/codilay/chunker.py`

**Chunker (Class):** This class handles the logic for splitting large files into smaller, processable chunks based on file structure or token count. It supports multiple programming languages and defines strategies for each to determine structural boundaries (e.g., class or function definitions in Python, JS/TS, Java, etc.). The chunking process is controlled by several parameters: 

- **Initialization:**
  - `token_counter`: A function to count tokens in a string, crucial for managing chunk sizes.
  - `config`: An instance of `CodiLayConfig`, providing customizable settings like `chunk_token_threshold`, `max_chunk_tokens`, and `chunk_overlap_ratio`.

- **Methods:**
  - `plan(file_path, content)`: Analyzes the given file content to create a `ChunkPlan`. If the total tokens are below a threshold, the entire content forms a single FULL chunk. Otherwise, it employs a multi-step process to generate a SKELETON chunk and multiple DETAIL chunks:
    - Skeleton extraction captures import statements, function/class signatures, and module-level declarations.
    - Boundary detection identifies key sections within the code to facilitate logical separation.
    - Detail chunks are created based on these boundaries or by token budgeting. Overlapping is added to aid in context retention across chunks.
  - `_extract_skeleton(lines, ext)`: Extracts high-level structural elements from the file to form a concise skeleton, often used for initial analysis.
  - `_find_boundaries(lines, ext)`: Detects structural boundaries across different programming languages, guiding how chunks are sectioned.
  - `_split_on_boundaries(lines, boundaries)`: Divides content into chunks based on detected boundaries.
  - `_split_by_tokens(lines)`: Provides a fallback for chunking by token count when structural boundaries are minimal.
  - `_add_overlap(chunks, lines)`: Ensures continuity between chunks by adding overlapping lines from the previous chunk to the current one. The overlap size is determined by a ratio and capped at 30 lines to maintain context while avoiding excessive repetition.
  - `_sub_split_chunk(chunk)`: Splits overly large chunks into smaller sub-chunks maintaining a balance between granularity and coherence.
  - Helper methods such as `_is_import_line`, `_is_signature`, `_is_comment`, `_is_docstring_start`, `_is_docstring_end`, `_is_decorator`, `_is_export`, and `_is_type_or_const` detect specific line types based on file extension, enabling language-specific processing.

This class plays a critical role in managing file complexity, particularly in scenarios like code review, static analysis, or refactoring, where understanding and manipulating code segments efficiently is crucial.

## Triage Implementation
> File: `src/codilay/triage.py`

The `triage.py` module is responsible for pre-flight file classification in CodiLay, operating in two primary modes: **fast** and **smart**. The fast mode relies on static patterns for near-instantaneous classification, whereas the smart mode utilizes a Language Learning Model (LLM) to analyze the file tree and make informed decisions about file processing priorities without examining the actual content.

### Key Classes and Functions:
- **`FileCategory`**: Enum class defining categories such as `CORE`, `SKIM`, and `SKIP` to classify files.

- **`TriageResult` Dataclass**: Stores results of the triage process, including file counts in each category, project type detection, and any relevant reasoning or warnings.

- **`fast_triage` Method**: Classifies files based on predefined directory and file extension patterns. Employs `ALWAYS_SKIP_DIRS`, `ALWAYS_SKIP_EXTENSIONS`, and `ALWAYS_SKIM_FILES`.

- **`smart_triage` Method**: Engages with the LLM to dynamically classify files based on the file tree input. If the LLM is unavailable or fails, it defaults to fast triage.

- **`_detect_project_type`**: Heuristically determines the project type from common indicators in file names (e.g., presence of `pubspec.yaml` implies a Flutter project).

- **`estimate_tokens_saved`**: Computes an estimate of token savings by skipping irrelevant files, contributing to overall efficiency.

- **Safety Mechanisms**: Provides a fallback mechanism to the fast mode and applies a safety net to prevent processing of known irrelevant file types.

### Integration
- Utilizes patterns cached within variables like `ALWAYS_SKIP_DIRS` to manage unwanted file scanning.
- The `Triage` class interfaces with LLM through a client, if provided.

This module plays a crucial role in preparing the CodiLay processing pipeline by assigning importance to files, thereby optimizing where the attention of the more resource-intensive smart triage should be focused.

→ See [LLM Prompt Templates for CodiLay] for details on prompts used in smart triage.

## .gitattributes Configuration
> File: `.gitattributes`

The `.gitattributes` file is used to configure Git's behavior with files in the repository. It specifies attributes for different file types, ensuring consistent handling across different environments and tools. Key configurations include:

- **Text Files Normalization**: `* text=auto` ensures all text files are normalized to use LF line endings. Common programming files such as `.py`, `.md`, `.json`, etc., are explicitly set to `text eol=lf` to enforce line ending standards.

## .gitignore Configuration
> File: `.gitignore`

The `.gitignore` file specifies untracked files for a Git repository. These files and directories are usually generated during the build or runtime process, and do not need to be version controlled. In CodiLay, the `.gitignore` includes:
